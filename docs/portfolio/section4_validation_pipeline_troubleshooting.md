# Section 4 Troubleshooting: AI Baseline 검토 과정에서 발견한 게시글 검증 파이프라인 경계 위험

## 문제 정의

이번 Spring Boot 마이그레이션에서는 기능 이관 속도를 높이기 위해 AI가 게시글 CRUD baseline을 먼저 구성했다.  
baseline은 목록/상세/생성/수정/삭제를 빠르게 연결하는 데는 효과적이었지만, **DTO 검증, 서비스 교차필드 검증, DB 제약의 책임 경계가 섞여 있는 구조적 위험**을 내포하고 있었다.

즉, 이 문제는 운영 중 우연히 터진 버그가 아니라,  
**AI가 빠르게 작성한 baseline을 Human Architect가 리뷰하는 과정에서 선제적으로 식별한 구조적 결함**이다.

---

## Before

AI baseline은 다음을 빠르게 해결했다.

- `/api/posts/` 목록/상세/생성/수정/삭제와 메타 조회 API 연결
- 프론트 계약 필드(`model_etc`, `category_etc`, `ai_response`)를 수용하는 요청 처리
- 서비스 레벨에서 일부 도메인 규칙(예: model-detail 조합) 1차 검증

이 baseline은 **기능 동작과 프론트 연동 가시화**라는 목적에 충분히 기여했다.

---

## Problem

Human Architect 리뷰에서 확인한 핵심 결함은 아래와 같았다.

- 입력이 `Map` 기반이라 형식/누락/범위 검증이 엔드포인트마다 흔들릴 수 있음
- 검증 책임이 단일 계층(Service)에 과도하게 집중되어 규칙 누락/중복 가능성 증가
- `satisfaction` 같은 핵심 필드의 DB 수준 안전장치가 약해 비정상 데이터 저장 여지 존재
- 모델/플랫폼/카테고리 조합 규칙이 명시적 파이프라인으로 고정되지 않아 유지보수 시 회귀 위험 증가

즉 baseline은 **기능은 동작하지만 장기적으로 정합성 drift가 생길 수 있는 구조**였다.

---

## 왜 이 문제가 중요한가

이번 프로젝트의 핵심은 단순 기능 이관이 아니라,  
**Spring 기반에서 검증 파이프라인(입력 계약/도메인 규칙/DB 무결성)을 아키텍처적으로 고정하는 것**이다.

검증 경계가 흐려진 상태를 방치하면,

- 신규 필드 추가 시 검증 누락이 발생하기 쉽고
- API 계약과 도메인 규칙의 불일치가 누적되며
- 테스트가 있어도 실제 영속 계층에서 무결성 보장이 약해진다

따라서 이는 단순 버그 대응이 아니라 **도메인 안전성 통제 문제**였다.

---

## After

Human Architect는 baseline을 그대로 확정하지 않고, 검증 책임을 3중 파이프라인으로 재정렬했다.

### 1) DTO 계층 (`@Valid`) 정착
- `PostUpsertRequest`를 도입해 필수값/길이/범위 규칙을 명시
- Controller 입력을 `Map`에서 `@Valid DTO`로 전환

### 2) Service 계층 교차필드 검증 강화
- 모델-플랫폼 일치 여부 검증
- `기타` 모델/카테고리 선택 시 `*_etc` 필수 규칙 명시
- 비활성/폐기 모델 차단 및 작성자 권한 검증 유지

### 3) DB 계층 무결성 보강
- `Post` 엔티티에 `satisfaction` CHECK 제약 추가
- 비정상 범위 데이터의 최종 저장 방어선 확보

---

## 테스트로 어떻게 증명할지

### 1. DTO 검증 테스트
- 제목 길이/만족도 범위 검증 실패 케이스 확인

### 2. 서비스 규칙 테스트
- 모델-플랫폼 불일치 차단
- `기타` 카테고리에서 `category_etc` 누락 차단
- 작성자 외 수정 차단(권한 검증)

### 3. 회귀 테스트
- 정상 payload 생성 성공
- 전체 테스트(`./gradlew test`) 통과로 기존 섹션 회귀 여부 확인

---

## 이번 사례에서 강조할 점

이 사례의 핵심은 “구현하다가 버그가 나서 수정했다”가 아니다.

- AI는 빠르게 CRUD baseline을 만들었다
- Human은 검증 책임 경계를 아키텍처 관점으로 재검토했다
- 그 결과 DTO/Service/DB 3중 방어 파이프라인으로 재설계했다
- 테스트로 규칙이 실제로 강제됨을 증명했다

즉, **AI는 속도를 만들고 Human Architect는 정합성과 무결성 기준을 고정한다**는 협업 원칙을 보여주는 사례다.

---

## 면접에서 어떻게 설명할지

> 게시글 CRUD를 AI로 빠르게 이관한 뒤, Human Architect 리뷰에서 검증 책임이 서비스에 과집중된 구조적 위험을 발견했습니다. 그래서 입력 계약은 DTO `@Valid`, 교차필드 규칙은 Service, 최종 무결성은 DB CHECK로 3중 방어를 재구성했고, 규칙 위반/권한 위반 테스트로 이를 검증했습니다.

### 답변 포인트
- 문제를 기능 버그가 아닌 검증 경계 결함으로 정의
- AI baseline과 Human Architect 역할 분리를 명확히 설명
- 리팩터링 결과를 테스트 근거와 함께 제시
