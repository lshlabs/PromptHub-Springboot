# Section 3 Troubleshooting: AI Baseline 검토 과정에서 발견한 사용자 권한/계약 경계 위험

## 문제 정의

이번 Spring Boot 마이그레이션에서는 기능 구현 속도를 높이기 위해 AI를 활용해 사용자 도메인 baseline 코드를 먼저 작성했다.  
이 baseline은 프로필/설정/세션/탈퇴 API를 빠르게 연결하는 데는 유리했지만, **아키텍처 관점에서 권한 경계와 응답 계약 정합성 위험을 내포한 구조**였다.

즉, 이번 문제는 운영 중 실제 장애가 먼저 발생해서 발견한 것이 아니라,  
**AI가 빠르게 작성한 baseline을 검토하는 과정에서 사람이 구조적 결함을 선제적으로 식별한 사례**다.

---

## Before: AI가 빠르게 작성한 초기 Baseline

AI는 프론트 연동과 기능 동작을 우선하기 위해 다음과 같은 구조의 baseline을 제안했다.

- `UserController`, `UserService`, `UserSession`, `UserSettings` 중심 사용자 API 골격 구성
- 프로필 조회/수정, 비밀번호 변경, 설정 조회/수정, 세션 조회, 계정 삭제 빠른 연결
- 인증 사용자 기반 self 흐름 확보
- 요약 응답(`summary`)은 placeholder 집계값으로 우선 계약 형태만 맞춤

이 구조는 **단일 요청 기준으로는 충분히 동작**했고, 프론트 호환 API를 빠르게 맞추는 데도 도움이 되었다.  
즉 baseline의 목적은 “완성형 권한/계약 설계”가 아니라 **빠른 기능 가시화와 이관 속도 확보**에 있었다.

---

## Problem: Architect 관점에서 확인한 숨겨진 구조적 결함

baseline을 리뷰하면서 가장 먼저 확인한 문제는 **self 권한 경계와 응답 계약 정확성 기준이 분리되어 있지 않다는 점**이었다.

### 핵심 구조적 위험
- self-only 정책이 Service 기준으로 중앙화되지 않으면 권한 판단이 엔드포인트별로 분산될 가능성이 있음
- `summary` 집계 필드(`total_views`, `total_likes_received`, `total_bookmarks_received`)가 placeholder일 때 API 계약이 사실상 오염될 수 있음
- 세션 종료 정책이 조회/종료 API와 분리 정의되지 않으면 세션 무효화 일관성이 약해질 수 있음

즉, baseline은 **“기능은 돌아가지만 장기적으로 권한/계약 정합성 drift가 생길 수 있는 구조”**였다.

이 문제는 단순 버그라기보다,  
**AI가 빠르게 만든 기능 중심 baseline이 권한 경계·계약 정확성 관점까지는 충분히 반영하지 못한 전형적인 구조적 한계**라고 판단했다.

---

## 왜 이 문제가 중요한가

이번 프로젝트의 목적은 단순히 Django 기능을 Java 문법으로 옮기는 것이 아니라,  
**Spring 기반 백엔드 구조에서 권한 경계와 API 계약의 안정성을 갖춘 재설계 경험을 쌓는 것**이다.

따라서 단일 요청 기준으로 동작한다는 이유만으로 baseline을 그대로 확정하면,
- 권한 누수 가능성이 유지보수 과정에서 커질 수 있고
- 잘못된 placeholder 계약이 프론트/백엔드에 고정될 수 있으며
- 이후 섹션(게시글/통계/캐시)과 결합될 때 데이터 신뢰성 문제로 확대될 수 있다

즉, 이 문제를 초기에 구조적으로 정리하는 것은 **단순 버그 수정이 아니라 아키텍처 통제의 문제**였다.

---

## After: Human Architect가 재설계한 방향

AI baseline을 그대로 확정하지 않고, 아래 기준으로 리팩터링 방향을 다시 잡았다.

### 1) 권한 규칙의 책임 위치 고정
- self API 권한 판단은 Service 계층에서 1차 처리
- Controller는 라우팅/입력 중심으로 축소
- Method Security는 반복 규칙이 필요한 지점만 보조 적용

### 2) 요약 응답 계약의 정합성 복구
- placeholder 집계 제거
- `count/sum` 기반 실제 집계로 계약 재정의
- 필드 의미를 도메인 기준으로 고정

### 3) 세션 관리 정책 명시화
- 활성 세션 목록과 종료 동작의 규칙 분리
- 단일 세션 종료(`key`)와 기타 세션 종료(`all=true`)를 명시적 유스케이스로 분리
- 종료된 세션은 `revokedAt` 기준으로 일관 처리

### 4) 테스트를 경계 검증 수단으로 사용
- 기능 동작 여부뿐 아니라 권한/계약/상태 전이 일관성을 검증
- 비인증 접근 차단, 타 세션 종료 차단, 집계 응답 정확성을 중심 테스트로 고정

즉, 사람의 역할은 AI가 만든 코드를 미세 수정하는 것이 아니라,  
**권한 경계와 응답 계약 기준을 먼저 정의하고 그 기준에 맞게 baseline을 재구성하는 것**이었다.

---

## 테스트로 어떻게 증명할지

### 1. 권한/보안 경계 테스트
- 비인증 사용자의 `/api/auth/profile/` 접근 차단 검증
- 세션 종료 시 현재 사용자 소유 세션만 조작 가능한지 검증

### 2. 계약 정합성 테스트
- `summary` 응답이 placeholder가 아닌 실제 집계값을 반환하는지 검증
- 프론트 사용 필드 호환성 유지 여부 확인

### 3. 상태 전이/세션 테스트
- 단일 세션 종료(`key`) 동작 검증
- 기타 세션 종료(`all=true`) 동작 및 종료 개수 검증
- 종료된 세션이 조회 대상에서 제외되는지 검증

---

## 이번 사례에서 강조할 점

이 사례의 핵심은 **“구현하다가 문제가 터져서 고쳤다”**가 아니다.

핵심은 다음과 같다.

- AI는 빠르게 사용자 API baseline을 만들었다
- 사람은 baseline을 그대로 신뢰하지 않고 권한/계약 경계를 검토했다
- 그 과정에서 self 권한 분산 위험과 placeholder 계약 위험을 선제적으로 발견했다
- 이후 권한 기준, 응답 계약 기준, 세션 정책 기준을 다시 정의했다

즉, 이 사례는  
**AI를 활용해 개발 속도를 높이되, 최종 권한 경계와 데이터 계약 책임은 사람이 진다**는 이번 프로젝트의 핵심 서사를 보여준다.

---

## 면접에서 어떻게 설명할지

> 이번 마이그레이션에서는 AI를 활용해 사용자 API baseline을 빠르게 만들었습니다. 다만 기능이 동작한다고 바로 확정하지 않고, self 권한 검증 책임이 분산될 위험과 summary 집계 계약의 부정확성을 구조적 결함으로 판단했습니다. 그래서 Service 중심 권한 경계, 실제 집계 기반 응답 계약, 세션 종료 정책을 재정의했고, 권한/계약/상태 전이 테스트로 이를 검증했습니다.

### 답변 포인트
- 문제를 “기능 버그”보다 **구조적 결함**으로 정의
- AI의 역할은 빠른 초안 작성
- 인간의 역할은 권한 경계/계약 기준 설계
- 리팩터링 후에는 반드시 테스트로 증명
